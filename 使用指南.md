# workflow-kit 使用指南

> 版本: v2.0 · 2026-05-18

`workflow-kit` 是一套可迁移的大模型自动化开发工作流。它把需求输入、结构化确认、技术方案、落地计划、任务拆分、实现记录、测试验证、构建验收和全链路复核固定成可恢复的状态机。

当前默认行为重点偏向 Java/Maven 项目：S8 构建产物默认校验 `target/*.jar`，Java 项目默认读取 `Java开发规范.md`。包内提供 `docs/.workflow/project_config.json` 模板，非 Java 项目安装后应覆盖构建产物和命令，并在项目入口文件里替换语言专项规范。

## 一、这是什么

你把 `workflow-kit` 安装到项目后，Codex 或 Claude Code 会按项目入口文件读取本地规范，不再靠临时口头约定工作。

它包含三层内容：

1. 工作流引擎：`docs/.workflow/`
2. 项目入口模板：`开发AGENTS.md`、`开发CLAUDE.md`
3. 人类说明文档：`INSTALL.md`、`使用指南.md`

业务项目自己的需求文档、Bug 记录、知识库内容、根目录 OpenSpec 历史 change 不属于通用包。

## 二、包内文件

```text
workflow-kit/
├── INSTALL.md
├── 使用指南.md
├── 开发AGENTS.md          # 安装到 Codex 项目时复制为 AGENTS.md
├── 开发CLAUDE.md          # 安装到 Claude Code 项目时复制为 CLAUDE.md
└── docs/
    ├── README.md          # docs 全局导航
    ├── 01-features/       # 空目录，安装后由脚本创建 feature
    ├── 02-bug-fix/        # 空目录，安装后由脚本创建 bugfix
    ├── 03-knowledge/      # 空目录，目标项目自行沉淀知识
    └── .workflow/
        ├── 工作流规范.md
        ├── Java开发规范.md
        ├── project_config.json
        ├── Obsidian文档规范.md
        ├── 知识库规范.md
        ├── bug修复规范.md
        ├── agents/
        ├── scripts/
        └── templates/
```

## 三、安装

```bash
cp -r workflow-kit/docs your-project/
cp workflow-kit/开发AGENTS.md your-project/AGENTS.md
cp workflow-kit/开发CLAUDE.md your-project/CLAUDE.md
cp workflow-kit/INSTALL.md your-project/
cp workflow-kit/使用指南.md your-project/
```

如果目标项目已有入口文件，合并模板内容，不要覆盖项目原有说明。通用模板负责工作流启动协议；项目覆盖层负责技术栈、构建命令、代码目录、PR 规范和业务知识入口。

## 四、安装验证

在目标项目根目录执行：

```bash
python3 docs/.workflow/scripts/init_feature.py --list
```

再确认入口文件包含关键路径：

```bash
grep -n "docs/README.md" AGENTS.md CLAUDE.md
grep -n "工作流规范.md" AGENTS.md CLAUDE.md
grep -n "context_packets.py" AGENTS.md CLAUDE.md
grep -n "docs/01-features" AGENTS.md CLAUDE.md
```

Java 项目再确认：

```bash
grep -n "Java开发规范.md" AGENTS.md CLAUDE.md
```

构建产物模板在 `docs/.workflow/project_config.json`：

```json
{
  "build": {
    "artifact_pattern": "target/*.jar",
    "artifact_label": "Jar",
    "build_command": "mvn -DskipTests package",
    "build_record_keyword": "Jar"
  }
}
```

Java/Maven 项目可以直接使用模板默认值。非 Java 项目安装后必须按项目实际产物覆盖这些字段，例如前端可改为 `dist/**/*`，Go 可改为 `bin/*`。

## 五、开始一个新需求

在 Codex 或 Claude Code 中直接说：

```text
我有一个新需求要做：……
```

或：

```text
我有一个 PRD 文档，需要按工作流开发，按规范开始。
```

主 Agent 会先确认输入模式：

- A：已有需求文档
- B：只有想法或口述描述

然后执行：

```bash
python3 docs/.workflow/scripts/init_feature.py "功能名"
```

脚本会在 `docs/01-features/Fxx-功能名/` 下创建状态文件、导航、恢复包、阶段目录和 feature-local OpenSpec 骨架。

## 六、人工锚点

流程中通常需要你在关键节点输入口令：

| 锚点 | 时机 | 口令 |
|------|------|------|
| ① | 需求输入脑暴完成 | `approve-req-input` |
| ② | 需求确认基线冻结 | `approve-req-final` |
| ③ | 技术方案疑问确认 | `approve-tech-question` |
| ④ | 全链路验证确认 | `approve-verify` |
| ⑤ | 最终验收发布 | `approve-release` |

## 七、会话恢复

中断后重新打开工具，说：

```text
恢复 F01 的开发
```

或手动执行：

```bash
python3 docs/.workflow/scripts/init_feature.py --recover F01
```

主 Agent 必须读取 `state.json`、恢复包和当前上下文包，输出恢复确认卡，然后等待你的下一步指令。

## 八、常用脚本

```bash
# 初始化新 feature
python3 docs/.workflow/scripts/init_feature.py "功能名称"

# 查看所有 feature
python3 docs/.workflow/scripts/init_feature.py --list

# 恢复指定 feature
python3 docs/.workflow/scripts/init_feature.py --recover F01

# 查看当前状态和合法动作
python3 docs/.workflow/scripts/stage_gates.py check F01

# 生成阶段上下文包
python3 docs/.workflow/scripts/context_packets.py build F01 S6 --task T01
python3 docs/.workflow/scripts/context_packets.py build F01 S8
python3 docs/.workflow/scripts/context_packets.py build F01 S10
python3 docs/.workflow/scripts/context_packets.py list F01

# 记录小步骤
python3 docs/.workflow/scripts/stage_gates.py step-start F01 "步骤名" '{"goal":"...","expected_outputs":["..."],"done_definition":["..."],"next_step":"..."}'
python3 docs/.workflow/scripts/stage_gates.py progress F01 '{"completed_action":"...","key_conclusions":["..."],"outputs":["..."],"verification":["..."],"next_step":"..."}'
python3 docs/.workflow/scripts/stage_gates.py step-done F01 "步骤名" '{"outputs":["..."],"key_conclusions":["..."],"next_step":"..."}'

# 子 Agent 返回建议携带 subagent-start 输出的 dispatch_id
python3 docs/.workflow/scripts/stage_gates.py subagent-done F01 "任务实现" '{"dispatch_id":"d-YYYYMMDDHHMMSS-xxxxxxxx","status":"done","summary":"...","output_paths":["..."],"key_conclusions":["..."]}'

# 校验器
python3 docs/.workflow/scripts/validators.py req_coverage F01
python3 docs/.workflow/scripts/validators.py fact_inheritance F01
python3 docs/.workflow/scripts/validators.py rdt_mapping F01
python3 docs/.workflow/scripts/validators.py impl_drift F01 T01
python3 docs/.workflow/scripts/validators.py rdtv_closure F01
```

```text

├── state.json
├── 导航.md
├── 恢复包.md
├── 00-需求输入/
├── 01-需求确认/

├── 02-技术方案/
│   ├── 代码影响点与依赖逻辑清单.md
│   └── 技术方案一致性检查.json

├── 04-实现记录/

```

`state.json` 是唯一权威状态文件。阶段开始、派遣子 Agent 前必须生成当前阶段上下文包。

## 十、迁移原则

迁移 `workflow-kit` 时，只带走通用流程资产：

- 带走 `docs/.workflow/`
- 带走入口模板和说明文档
- 带走空的 `docs/01-features/`、`docs/02-bug-fix/`、`docs/03-knowledge/`

不要带走来源项目已有 feature、bug、knowledge、业务 `CLAUDE.md` 或根目录 OpenSpec 历史 change。

## 十一、常见问题

**非 Java 项目能用吗？**

能。流程规范是语言无关的，但默认构建门禁偏向 Java/Maven/Jar。非 Java 项目可以保留 `Java开发规范.md` 但不默认读取，也可以替换为对应语言规范，并必须覆盖 `docs/.workflow/project_config.json` 的构建产物路径和命令。

**目标项目已有 AGENTS.md 或 CLAUDE.md 怎么办？**

合并，不覆盖。保留项目原有构建、测试、目录结构、PR 和安全要求，把 workflow-kit 的启动协议、核心脚本、上下文加载协议和严格禁止事项合并进去。

**docs 目录要不要进 Git？**

建议 `docs/.workflow/` 进 Git。feature、bug、knowledge 文档是否进 Git，由目标项目决定。入口模板要求大模型不得自动 `git add` 或提交这些工作流文档，除非人工明确要求。

**OpenSpec 放在哪里？**

feature-local OpenSpec 是权威位置：`docs/01-features/Fxx-功能名/openspec/`。根目录 `openspec/` 可以作为 CLI 镜像或全局能力规范，但不要把业务历史 change 混入通用包。
