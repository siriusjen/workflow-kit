## A. 安装模型:从"覆盖"改为"合并"(最高优先级)

### 证据
- `INSTALL.md:11-12,36-37`、`使用指南.md:16,27-28,50-51`、`工作流规范.md:1001-1002` 让用户 `cp workflow-kit/开发AGENTS.md your-project/AGENTS.md`。
- 仓库实际只有根 `AGENTS.md` / `CLAUDE.md`,**不存在 `开发AGENTS.md` / `开发CLAUDE.md`** → 首条命令报 `No such file or directory`。
- 即便存在,`cp` 也会**覆盖**目标项目已有入口文件,抹掉项目自身内容(目标项目几乎都已有自己的 AGENTS.md/CLAUDE.md)。

### 方案(复用 OpenSpec 受管块范式)
根 `AGENTS.md` 顶部已有 `<!-- OPENSPEC:START --> … <!-- OPENSPEC:END -->` + "保留此块以便 `openspec update` 刷新"。完全照搬此模式:把工作流入口正文包进 `WORKFLOW-KIT` 标记,提供幂等注入脚本。
- 整体复制类:`docs/.workflow/`、空的 `docs/01-features|02-bug-fix|03-knowledge`、`docs/README.md`。
- 合并注入类:`AGENTS.md` / `CLAUDE.md` 的 `WORKFLOW-KIT` 受管块。

### 步骤
1. **A-D1 加标记**:在两份根入口文件,把"工作流正文"(`# 开发工作流入口` → `## 严格禁止` 结束)外包一对
   `<!-- WORKFLOW-KIT:START -->` / `<!-- WORKFLOW-KIT:END -->`。
   - `## 项目覆盖层` 留在标记**外**(目标项目自有内容承担此角色,不注入)。
   - 块尾(END 前)加一行:`> 保留此受管块,python3 docs/.workflow/scripts/install_entry.py 会据此刷新工作流入口内容。`
   - 两文件块内文字保持各自现状(AGENTS 面向 Codex、含 Bug Skill 调度表;CLAUDE 面向 Claude Code),只加标记与说明行。
2. **A-D2 新增 `docs/.workflow/scripts/install_entry.py`**(纯标准库,`Path(__file__).parent` 定位):
   - 用法:`--target <项目根>`;可选 `--only AGENTS.md|CLAUDE.md`、`--dry-run`。
   - 源:从 kit 根入口文件抽取 `START…END`(含标记);源缺标记→报错退出提示先做 A-D1。
   - 目标:无文件→创建仅含受管块;有标记→整段替换两标记间内容(含标记);无标记→末尾追加(原内容保留)。
   - **标记外字节绝不改**;幂等(连跑两次结果一致,不重复、不堆空行);`--dry-run` 打印 unified diff;成功 0、错误非 0 且原因明确。
   - 参考核心:
     ```python
     START, END = "<!-- WORKFLOW-KIT:START -->", "<!-- WORKFLOW-KIT:END -->"
     def upsert(target, block):
         if START in target and END in target:
             return target[:target.index(START)] + block + target[target.index(END)+len(END):]
         sep = "" if target == "" or target.endswith("\n\n") else "\n\n"
         return target + sep + block + "\n"
     ```
3. **A-D3 重写安装文档**(`INSTALL.md`、`使用指南.md`、`工作流规范.md §11`):删除全部 `开发AGENTS/开发CLAUDE` 字样,安装改两步:
   ```bash
   cp -r workflow-kit/docs/.workflow your-project/docs/.workflow
   cp -n workflow-kit/docs/README.md your-project/docs/README.md
   mkdir -p your-project/docs/{01-features,02-bug-fix,03-knowledge}
   python3 your-project/docs/.workflow/scripts/install_entry.py --target your-project
   ```
   保留"手工粘贴受管块"作为降级说明。
4. **A-D4** 在 `使用指南.md`/`README` 明确"整体复制类 vs 合并注入类"两类资产。
5. **A-D5 测试 `tests/test_install_entry.py`**:无文件→创建;有自定义内容+无标记→保留并追加;有旧块→整段替换无重复(幂等连跑两次一致);`--dry-run` 不写;源缺标记→报错非 0。

### 影响面
文档 3 处 + 入口文件加标记 + 新增 1 脚本 1 测试。无 state/门禁语义变更。

### 验收
干净环境照新文档全程零报错;自定义入口内容逐字保留;连跑两次幂等;`grep -rn "开发AGENTS\|开发CLAUDE" .` 零命中;两套测试全绿。

## H. 补回归测试(脚本重构前置安全网)

### 验收
重构前后跑同一组测试结论一致;新测试与现有测试全绿。

## B. 阶段/门禁/子Agent 定义抽到单一配置(重构地基)

### 证据
`FEATURE_STAGE_CANONICAL`/`FEATURE_STAGE_DISPLAY` 在 stage_gates、context_packets、init_feature、validators **各存一份**;`KNOWN_SUBAGENTS`/`SUBAGENTS_BY_STAGE`、`STAGE_DEFS`/`BUG_STAGE_DEFS` 也硬编码在脚本内。改一处要同步 4 个文件,极易漂移,且新项目增删阶段必须改 Python。

### 方案
下沉到 `docs/.workflow/workflow_definition.json`(声明式),新增 `workflow_def.py` 加载器,运行时读取,单一事实源。

### 步骤
1. 设计 schema:`stages[]`(canonical/display/order)、`gates[]`(key + 后续 D 用的 prereq/validator 声明)、`subagents[]`(name/stage)、`stage_context[]`(must_read/on_demand 模式)。
2. 新增 `workflow_def.py`(无第三方依赖),对缺键/类型错误明确报错。
3. 4 个脚本把硬编码常量替换为加载结果(保留同名常量作为加载缓存,最小化调用点改动)。

### 影响面
4 脚本 + 1 配置 + 1 加载器。逻辑等价重构,不改门禁语义。

### 验收
删除任一脚本内重复常量后,用 demo feature 跑 S1→S10,state.json 演化与改造前完全一致;加载器对残缺配置报错;H 的测试全绿。

## C. 校验器自动挂到门禁(反转控制流)

### 证据
规范要人工先 `validators.py rdt_mapping <FID>` 再 `auto rdt-mapping-passed`;而 `validators.py` 的 `_trigger_auto` 反过来 shell 调 `stage_gates.py auto`,两者互调易不一致;真实使用极易漏跑校验器导致门禁卡死。

### 方案
`stage_gates.py auto <gate>` 内部按门禁绑定自动调用校验器,通过才转移,失败原样阻断并输出缺口。人工不再手动跑校验器。

### 步骤
1. 在 B 的 `gates[]` 给每个门禁声明 `validator`(可空)。
2. 把 validators 的检查改为可 import 的函数(返回结构化结果),`auto` 前按声明调用。
3. 移除 validators 对 stage_gates 的反向 shell 调用。
4. 保留 `validators.py <name> <FID>` 命令行入口仅供人工只读排查(不再触发转移)。

### 影响面
`stage_gates.py` auto 分支 + `validators.py` 函数化。依赖 B。

### 验收
对缺 R→D→T 映射的 feature 执行 `auto rdt-mapping-passed`→不转移并打印缺口;补全后再执行→转移。覆盖 `req-cross-validated`/`artifact-package-done`/`http-acceptance-done`。

## D1. `validate_auto_prereqs` 改为声明式前置

### 证据
`stage_gates.py` 的 `validate_auto_prereqs` 为约 125 行 if/elif,靠 glob+正则在 markdown 找证据(如 `RED|失败测试|红绿`、`passed: true/通过`),命名一变即静默失效;加门禁=加分支+正则,难测。

### 方案
每个门禁在配置声明前置类型(`file_glob`/`keyword_in_file`/`checklist_true`/`state_field`),脚本用统一循环求值,函数降到约 10 行。

### 步骤
1. `gates[]` 增 `prereqs[]`。
2. 写通用 prereq 求值器,聚合缺口信息。
3. 现有 if/elif 逐条翻译为声明并删除分支(先用 H 的"快照对照"测试兜底)。

### 影响面
`stage_gates.py` 单文件。依赖 B。

### 验收
所有现有门禁改造前后判定逐一相同;新增门禁只改配置不改代码。

## D2. 关键元数据升为 state.json 一等字段

### 证据
docs-only 标记、worktree 是否创建/`code_worktree_path`、build 配置覆盖等,靠在 markdown 文本里正则扒取来判断门禁,脆弱且可能与状态不一致。

### 方案
`docs_only`、`worktree_created`、`build_config_overridden` 等成为 state.json 显式字段,由对应 `auto`/`exception` 动作写入,门禁只读 state。

### 步骤
1. 扩展 state schema(向后兼容:缺字段按默认值)。
2. `auto docs-only-*`、`worktree-created`、`exception worktree` 等动作显式写字段。
3. 门禁改读字段,文本扒取仅作迁移期回退。

### 影响面
`stage_gates.py` + 规范 §10 schema 文档。依赖 B/D1。

### 验收
不依赖 markdown 正则即可正确放行/阻断 docs-only 与 worktree 门禁;docs-only 三件套端到端通过。

## E. 上下文包模板化 + 缺口自动发现

### 证据
`context_packets.py` 的 `build_packet` 用 8 段字符串拼接;`expand_must_read_items` 找不到文件时塞"（待生成或待定位）"占位,需人工再去手找。

### 方案
- 包正文用模板渲染(模板放 `templates/上下文包.md.tpl`)。
- 未命中的 must-read 自动与当前 `in_progress_step.done_definition`/`expected_outputs` 对照,输出结构化"待生成清单"而非笼统占位。

### 步骤
1. 抽模板,build 时填充变量。
2. `expand_must_read_items` 对缺口结合当前步骤期望产物给出明确列表。

### 影响面
`context_packets.py` + 新模板。

### 验收
包格式调整只动模板;缺产物时包内显式列"待生成: <路径>（来自 done_definition）";S6(`--task T01`)与 S8 各生成一次断言关键段落与缺口提示存在。

## F1. 抽公共工具模块去重

### 证据
`load_json_with_snapshot_fallback` 在 init_feature 与 stage_gates 各一份;子Agent 过滤、step-start/progress/step-done 三套结构相同的 JSON 校验等多处重复。

### 方案
新增 `workflow_common.py`,集中:快照恢复、子Agent 过滤、step JSON 校验、state 保存(统一更新 `last_updated`)。逐函数迁移,签名不变以缩小 diff。

### 影响面
全部脚本,纯重构。可与 B 合并推进。

### 验收
重复函数仅存一份,行为不变;现有+新增单测全绿。

## F2. 统一配置 + schema 校验

### 证据
`workflow_config.py` 只管 4 项;build 配置又在 stage_gates 以 `DEFAULT_BUILD_CONFIG` 另存;`project_config.json` 无 schema 校验,缺键到运行时才报错。

### 方案
`build` 与 `workflow` 配置统一进 `project_config.json`,加载时 schema 校验并列出全部可配置键(可发现性);缺失给默认+告警,类型错直接报错。

### 影响面
`workflow_config.py` + stage_gates build 读取点 + 文档。

### 验收
删/错配任一键,加载即明确提示;文档列全部可配置项及默认值。

## G. 降低人工簿记摩擦(实用性)

### 证据
S1–S10 全程手敲 `step-start`/`progress`/`step-done`/`subagent-start`/`subagent-done`,参数为大段 JSON,`dispatch_id` 需人工回填,极易漏记导致门禁卡死。

### 方案(不削弱并行/同名场景强校验)
1. `subagent-done` 不传 `dispatch_id` 时,若当前步骤仅一条未闭合派遣→自动匹配;多条仍强制显式。
2. `step-done` 省略步骤名时默认取当前 `in_progress_step.step`。
3. `next_step` 给合理默认;文档补"最短调用"示例。

### 影响面
`stage_gates.py` 入参处理 + 文档。

### 验收
单派遣可省 `dispatch_id`;多派遣仍强制显式且缺失即失败;现有强校验测试不回退。

## I. 文档去 Java 偏向叙事

### 证据
README/使用指南默认以 Jar/Maven 叙事,前端/Go 项目首次接触易被误导(虽已可经 project_config 覆盖)。

### 方案
把"流程语言无关 + 默认门禁偏 Java、需经 `project_config.json` 覆盖"提到 README/使用指南更靠前位置,给出前端(`dist/**/*`)、Go(`bin/*`)覆盖示例。

### 影响面
纯文档。

### 验收
非 Java 读者在开头即明白如何覆盖构建门禁。
