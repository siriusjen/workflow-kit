# F01 工作流自动化优化 需求说明书 v1

> 日期: 2026-05-29
> 输入来源: `00-需求输入/01-原始方案-工作流自动化优化20260529.md`
> OpenSpec 模式: formal-change

## 目标

把 workflow-kit 从“靠人工复制、人工记门禁、脚本多处硬编码”的状态，改造成可迁移、可验证、低簿记摩擦的自动化工作流基线。落地必须保持 S1-S10 阶段数量、门禁顺序和现有 state.json 语义不破坏。

## 需求清单

### R01 安装入口必须从覆盖改为合并注入

当前 `INSTALL.md`、`使用指南.md` 和 `工作流规范.md` 引用不存在的 `开发AGENTS.md` / `开发CLAUDE.md`，且安装命令会覆盖目标项目已有入口文件。必须改为 `WORKFLOW-KIT` 受管块合并注入。

验收:
- `AGENTS.md` / `CLAUDE.md` 中工作流入口正文被 `<!-- WORKFLOW-KIT:START -->` 与 `<!-- WORKFLOW-KIT:END -->` 包裹。
- 目标入口文件已有内容逐字保留，受管块可新增或替换。
- 连续运行安装注入命令结果幂等。
- 仓库中不再出现 `开发AGENTS` 或 `开发CLAUDE` 安装指引。

### R02 安装脚本来源必须可迁移

安装脚本在目标项目内运行时，不能隐式依赖仍可访问 workflow-kit 源根目录。受管块模板必须随 `.workflow` 迁移，或脚本必须提供明确源路径参数。

验收:
- 干净目标目录只复制 `docs/.workflow` 后，脚本仍能注入入口文件。
- `--dry-run` 输出 unified diff 且不写文件。
- 源模板缺失、标记不完整、目标标记不完整时输出明确错误并非 0 退出。

### R03 脚本重构前必须补行为回归测试

在重构阶段表、门禁和校验器前，必须补足关键行为基线测试，避免状态机语义漂移。

验收:
- 覆盖安装注入、人工锚点、docs-only 三件套、worktree 门禁、artifact/http 门禁失败路径、bug 链阻断。
- 新旧测试均通过。

### R04 阶段、门禁、子 Agent 定义必须收敛为单一事实源

`FEATURE_STAGE_CANONICAL`、`FEATURE_STAGE_DISPLAY`、`KNOWN_SUBAGENTS`、`SUBAGENTS_BY_STAGE`、`STAGE_DEFS`、`BUG_STAGE_DEFS` 等不得继续在多个脚本中漂移。

验收:
- 新增 `docs/.workflow/workflow_definition.json` 和加载器。
- 现有脚本从加载器读取等价定义。
- 缺键或类型错误时加载器明确报错。

### R05 校验器必须由门禁自动触发

`validators.py` 不得在校验通过后反向 shell 调用 `stage_gates.py auto` 作为主要路径。`stage_gates.py auto <gate>` 应根据门禁定义自动执行绑定校验器。

验收:
- 缺 R-D-T 映射时执行 `auto rdt-mapping-passed` 不转移并打印缺口。
- 补齐映射后同一命令可完成转移。
- `validators.py <name> <FID>` 保留为只读排查命令，不推进 state。

### R06 自动门禁前置条件必须声明式

`validate_auto_prereqs` 中大量 if/elif 和 Markdown 正则应迁移为配置声明和统一求值器。

验收:
- 每个门禁的前置条件在配置中声明。
- 新增门禁优先只改配置。
- 现有门禁改造前后判定一致。

### R07 关键运行元数据必须进入 state.json 一等字段

docs-only、worktree、构建覆盖等关键判断不得只靠 Markdown 正则扒取。

验收:
- 新字段向后兼容，缺失时有默认值。
- `auto docs-only-*`、`auto worktree-created`、`exception worktree` 等动作显式维护运行元数据。
- 迁移期保留文本证据回退，但门禁优先读 state。

### R08 上下文包必须模板化并输出明确缺口

`context_packets.py` 不应把包正文写死在 Python f-string 中；must-read 缺失时应输出结构化待生成清单。

验收:
- 包格式调整只需修改模板。
- S6/S8 生成包时能列出来自 `expected_outputs` 和 `done_definition` 的缺口。

### R09 公共工具与配置必须去重并校验

快照恢复、状态保存、步骤 JSON 校验、配置读取等公共能力应集中到共享模块；`project_config.json` 应有 schema 校验和默认值说明。

验收:
- 重复函数收敛到公共模块。
- build/workflow 配置统一从 `project_config.json` 加载。
- 类型错误直接报错，缺失键给默认值和可理解提示。

### R10 人工簿记命令必须降低摩擦但保持强校验

常见单子 Agent、当前步骤完成等场景允许省略重复参数；并行或同名派遣仍必须显式 disptach_id。

验收:
- 单一未闭合派遣可省略 `dispatch_id`。
- 多条未闭合派遣省略 `dispatch_id` 必须失败。
- `step-done` 可省略步骤名并默认取当前步骤。

### R11 文档叙事必须弱化 Java 默认误导

README/使用指南/安装说明应明确 workflow-kit 语言无关，Java/Maven/Jar 只是默认配置，可通过 `project_config.json` 覆盖。

验收:
- 非 Java 读者在安装和使用开头即可看到覆盖方式。
- 提供前端和 Go 的构建配置示例。

## 非目标

- 不改变 S1-S10 阶段数量和门禁顺序。
- 不迁移历史 feature、bugfix、knowledge 业务记录。
- 不把根目录 OpenSpec 历史 change 作为本 feature 的权威文档。
- 不自动暂存或提交 docs 下的 workflow 过程文档。
