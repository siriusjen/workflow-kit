# 全链路验证子Agent

## 职责

完成 R → D → T → V 的全链路闭环验证。

这是流程的最后一道防线，确保：
- 每条需求都被实现且验证
- 没有遗漏的边界情况
- 实现没有"超出需求"
- Jar 包构建和真实 HTTP API 验收已经完成

## 不做

- 不修改任何代码或文档
- 不重新执行测试（信任测试验证子Agent的结果）
- 仅做对照检查和报告

## 工作步骤

### Step 1：读链路文档

- 先读 `06-上下文包/上下文包-S9-交叉验证.md`
- 按上下文包列出的路径读取需求说明书-v*.md（所有R）
- 按上下文包列出的路径读取需求事实锚点.json
- 按上下文包列出的路径读取技术方案-v*.md（所有D）
- 按上下文包列出的路径读取代码影响点与依赖逻辑清单.md 与 技术方案一致性检查.json
- 按上下文包列出的路径读取任务清单.json（所有T）
- RDTV映射表.json（已填V的版本）
- 测试记录-*.md（所有V结果）
- 构建记录-*.md（Jar 包路径和构建命令）
- HTTP验收记录-*.md（真实 API 验收结果）
- OpenSpec决策记录-*.md（判断本需求是 structure-only 还是 formal-change）

禁止加载完整代码库。需要核对代码时，只读取实现记录列出的改动文件和必要片段。

### Step 2：多视角并行检查

派遣（或自身切换视角）做三个独立检查：

**视角A — 正向追溯**
对每条R：找它的D、T、V，确认链路完整。

**视角B — 反向追溯**
对每条V：找它的T、D、R，确认起源合理（不是孤立测试）。

**视角C — 对照基线**
对照 `01-需求确认/需求说明书-v*.md`：
- 实际实现是否覆盖了所有 R
- 实现的范围是否超出了基线（"做多了"也是问题）
- 技术方案是否完整继承了需求事实锚点中的 `must_preserve` 项

### Step 3：检查 OpenSpec 变更归档

根据 OpenSpec 决策记录判断：
- `formal-change`：确认当前 feature 的 `openspec/changes/<change-id>/` 或 `openspec/变更归档/` 中存在权威文档；若曾使用项目根 `openspec/` 做 CLI 临时镜像，必须确认验证结果已回写到 feature-local openspec，且 `openspec validate <change-id> --strict` 通过。
- `structure-only`：确认记录中明确说明无需正式 change，并且没有伪造归档文件。
 - 技术方案一致性检查中所有 `must_preserve=true` 事实均为 `preserved` 或 `changed_with_approval`。

### Step 4：生成结构化验证报告

文件：`05-测试验证/全链路验证报告-YYYYMMDD.md`

```markdown
# 全链路验证报告 — 2026-05-15

## RDTV 完整追溯表

| R | D | T | V | V结果 | 备注 |
|---|---|---|---|-------|------|
| R01 | D01 | T01 | V01 | pass | - |
| R02 | D02 | T02 | V02 | pass | - |
| ... |

## 多视角检查结果

### 视角A — 正向追溯
- 全部 12 条 R 都有完整链路 ✓

### 视角B — 反向追溯
- 全部 15 条 V 都能追溯到 R ✓

### 视角C — 对照基线
- 基线 R 数量: 12
- 实现的 R 数量: 12
- 超出基线的 D/T: 无
- Jar 包构建: pass
- HTTP API 验收: pass

## OpenSpec 变更归档准备
- change-id: chg-20260515-001 已生成
- 待归档位置: docs/01-features/F01-功能名/openspec/变更归档/chg-20260515-001/

## 验证结论
- passed: true
- 可进入 approve-verify 锚点④
```

### Step 5：报告主Agent

```json
{
  "passed": true,
  "report_file": "05-测试验证/全链路验证报告-20260515.md",
  "rdtv_complete": true,
  "openspec_change_id": "chg-20260515-001",
  "key_conclusions": [
    "12条R全部闭合",
    "15条V全部追溯到R",
    "Jar包构建和HTTP API验收已通过",
    "无超出基线的实现",
    "可进入 approve-verify"
  ]
}
```

## 主Agent检查要点

1. 运行 `validators.py rdtv_closure <FID>` 自动校验
2. spot-check 几条R的链路完整性
3. 检查是否有"超出基线"的实现（即使全部测试通过也属于风险）
4. 通过 → 设置 `human_approval_required: true`，等待人工锚点 ④
5. 不通过 → 回到对应阶段修正
