## 2. 命名规范

### 2.1 核心原则

- **编号在前，名称在后**：`01-名称`、`BF01-名称`、`F01-名称`
- **见名知意**：看文件名不需要打开就知道内容
- **中文优先**：面向团队协作，文件名用中文业务语言
- **不用空格**：用连字符 `-` 分隔编号和名称

### 2.2 编号体系

| 前缀 | 适用场景 | 示例 |
|------|----------|------|
| `数字-` | 固定目录的顺序编号 | `01-环境配置`、`02-架构设计` |
| `F{nn}-` | Feature 功能编号 | `F01-示例功能`、`F12-权限管理` |
| `BF{nn}-` | Bug Fix 编号（当天内顺序） | `BF01-提交500错误`、`BF02-额度计算错误` |
| `R{nn}-` | 需求编号（在 feature 内部） | `R01-提交申请`、`R05-工作日计算` |
| `D{nn}-` | 设计方案编号 | `D01-状态机设计` |
| `T{nn}-` | 任务编号 | `T01-创建数据表` |

### 2.3 文件名规范

```
格式：{编号}-{业务名称}.md

示例：
  ✓ 需求说明书-v1.md
  ✓ 技术方案-v2.md
  ✓ 落地计划-v1.md
  ✓ BF01-示例接口保存500错误-总览.md

  ✗ requirement.md          （无编号，英文，无业务语义）
  ✗ 新建文档.md             （无编号，无业务语义）
  ✗ doc1.md                 （无意义编号）
  ✗ 技术方案 v2.md          （空格分隔）
```

### 2.4 版本号规范

文档有版本演进时在文件名末尾加 `-v{n}`：

```
需求说明书-v1.md   ← 初版
需求说明书-v2.md   ← 修订版（保留 v1，不覆盖）
需求说明书-v3.md   ← 再次修订
```

**不允许**在同一文件上直接覆盖修改后当做"新版本"，版本历史通过保留旧文件体现。

## 3. 文档头部规范（每个文档必须有）

每个 `.md` 文件的头部必须包含以下信息：

```markdown
# {文档标题}

> **所属**: F01-示例功能 / 01-需求确认
> **版本**: v1.0
> **创建时间**: 2026-05-15
> **最后更新**: 2026-05-15
> **状态**: 草稿 / 评审中 / 已确认 / 已归档
> **关联**: [[需求说明书-v1]] · [[技术方案-v1]]

---
```

**关联链接**：用 Obsidian 的 `[[文件名]]` 语法，自动生成双向链接，方便在图谱视图中追踪文档关系。

---

## 4. Obsidian 标签规范

文档头部可以加标签，方便全局搜索和过滤：

```markdown
#feature/F01   ← 属于哪个 feature
#stage/需求    ← 所处阶段（需求/方案/实现/测试/验收）
#status/进行中 ← 状态（草稿/进行中/已完成/已归档）
#bugfix/BF01   ← 属于哪个 bug fix
#type/规范     ← 文档类型（规范/记录/分析/方案）
```

## 5. 各目录的文档创建规则

### 5.1 01-features/（功能开发）

由 `init_feature.py` 脚本自动创建，遵循工作流规范。
**不要手动在这里创建文件**，由大模型按流程自动生成。

```
├── 04-实现记录/

```

### 5.1.1 上下文包目录规则

`06-上下文包/` 只存放阶段级最小上下文包，由 `context_packets.py` 生成或更新。

命名格式：

```
上下文包-S{阶段号}-{阶段名}.md
```

上下文包必须是“索引 + 摘要 + 按需加载清单”，不得粘贴完整需求全文、完整代码、完整测试日志。它用于给主Agent和子Agent提供最小启动上下文，并通过路径指向真实事实来源。

### 5.1.2 Git 与 OpenSpec 归档规则

- `docs/` 下的 feature 文档默认由工作流生成，但大模型不得自动 `git add`、暂存或提交；是否纳入 Git 由人工决定。
- `formal-change` 的权威 OpenSpec 文档统一放在对应 feature 的 `openspec/changes/<change-id>/` 下，完成后归档到 `openspec/变更归档/`。
- 若项目根 `openspec/` 仅用于 OpenSpec CLI 校验或历史兼容，它不是 feature 文档的最终归档位置。

### 5.2 02-bug-fix/（Bug 处理）

详见 `docs/.workflow/bug修复规范.md`。

### 5.3 03-knowledge/（知识库）

详见 `docs/.workflow/知识库规范.md`。

不要把临时笔记、未完成的内容放在 knowledge 里，只放**稳定可复用的信息**。

### 5.4 04-retrospective/（复盘，可选）

按月归档，文件名格式：`{YYYY-MM}-{主题}-复盘.md`

## 6. README.md 维护规范

`docs/README.md` 是整个 docs 的统一索引，**每次新增重要文档都需要在这里更新**。

由大模型在以下时机自动更新：
- 新建 feature（init_feature.py 执行时）
- Bug Fix 创建时
- 知识库新增文档时

人工在以下时机手动更新：
- 添加外部参考链接
- 标记某个功能为"已归档"

---

## 7. 禁止事项

- ❌ 在 `docs/` 根目录下直接创建散落文件（所有文件必须在有编号的子目录下）
- ❌ 无编号的目录名（`feature/`、`bugs/`、`notes/`）
- ❌ 无意义的文件名（`新建文档.md`、`doc1.md`、`temp.md`）
- ❌ 直接覆盖旧版本文档（用版本号保留历史）
- ❌ 在 `.workflow/` 目录下手动创建文档（这是大模型的工作区）
