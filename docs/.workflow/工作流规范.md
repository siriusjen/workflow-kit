```
              approve-tech-question / approve-verify /

```

默认规则：
- 除非在修订规范、审计全流程，或当前问题横跨多个阶段，否则**不要整篇读取本文**。
- 主Agent 启动时只读速查卡；进入具体阶段后，优先读当前阶段上下文包，再按上表补读最小章节。
- 子Agent 默认不读本文全文；只在其定义文件要求时读取速查卡或指定小节。
- 入口文件（`AGENTS.md` / `CLAUDE.md`）中的“读规范”均按本表解释为读取索引、速查卡、最小加载地图或明确小节，不代表全文加载。

## 1. 工具定位

### 1.1 OpenSpec — 结构化推理引擎
- **本质**：把模糊输入变结构化规范
- **用在**：需求结构化、正式能力变更提案、技术方案草稿、验证阶段一致性复核
- **不用于**：写代码、执行测试、自由发散讨论
- **双模式**：
  - `structure-only`：仅把需求转成 R 编号、场景、验收边界，并写 `01-需求确认/OpenSpec决策记录-YYYYMMDD.md`。
  - `formal-change`：当涉及新增能力、破坏性 API/数据结构、架构模式、安全/性能语义变化时，必须按 `openspec/AGENTS.md` 的结构在当前 feature 下创建 `openspec/changes/<change-id>/`，即 `docs/01-features/Fxx-功能名/openspec/changes/<change-id>/`。若 OpenSpec CLI 只能识别项目根 `openspec/`，可临时镜像到根目录执行 `openspec validate <change-id> --strict`，但最终权威文档必须回写到 feature-local openspec。

OpenSpec 模式必须在 S2 记录，执行：

```bash
python3 docs/.workflow/scripts/stage_gates.py auto <FID> openspec-decision-recorded
```

未记录 OpenSpec 决策，禁止 `approve-req-final`。

### 1.2 Superpowers — 执行纪律引擎
- **本质**：有规范后高质量执行
- **用在**：想法发散（brainstorming）、澄清补充、落地计划、编码实现、测试验证
- **不用于**：把混乱需求结构化（应先用 OpenSpec）

### 1.3 串联原则
```
OpenSpec（结构化） → 人工确认 → Superpowers（执行）
```
两者不竞争、不替代。错误用法：让 Superpowers 直接结构化原始需求。

### 1.4 阶段技能矩阵（强制）

Superpowers 是技能，不是业务子Agent；OpenSpec 是规格/结构化工具，不是执行 Agent。主Agent 必须按下表选择，不允许临场自由替换。

| 阶段 | OpenSpec 使用 | Superpowers 技能 | 子Agent | 说明 |
|------|---------------|------------------|---------|------|
| 启动/恢复 | 不使用 | `using-superpowers`（会话级）；必要时 `strategic-compact` | 无 | 只读 `state.json`、恢复包和当前上下文包，不做需求推理 |
| S1 需求输入 | 不使用 | `brainstorming` | 无 | 用于发散和澄清；不生成正式 R/D/T |
| S2 需求确认 | 必须使用：`structure-only` 或 `formal-change` 决策 | 不强制使用 | `需求交叉验证` | OpenSpec 负责结构化需求和变更模式判定；子Agent只做输入/确认对照，并产出需求事实锚点 |
| S3 技术方案 | 读取 S2 的 OpenSpec 决策；`formal-change` 时对照 delta spec | 不强制使用 | `技术方案设计` | 不用 Superpowers 直接替代技术方案子Agent，且必须输出代码影响点与一致性检查 |
| S4 落地计划 | 不使用 | `writing-plans` | `落地计划` | `writing-plans` 提供计划纪律，子Agent产出项目内 `03-落地计划/落地计划-v1.md` |
| S5 任务拆分 | 不使用 | 不强制使用 | `任务拆分` | 任务拆分依赖 R/D/计划映射和 `validators.py rdt_mapping`，不是 OpenSpec 工作 |
| S6 实现 | 不使用 | `using-git-worktrees`、`subagent-driven-development`、`test-driven-development`；无子Agent能力时才用 `executing-plans` | `任务实现` | 每个 T 独立上下文、TDD、实现记录按 T 增量写入 |
| S7 测试验证 | 不使用 | `verification-before-completion` | `测试验证` | 先拿测试证据再声明通过，HTTP 验收不在本阶段做 |
| S8 构建验收 | 不使用 | `verification-before-completion` | `HTTP接口验收` | Jar 构建、HTTP 验收清单、真实 HTTP 请求都要有证据 |
| S9 全链路验证 | 使用：一致性复核；`formal-change` 时对照 OpenSpec change | `verification-before-completion` | `全链路验证` | 对照 R-D-T-V、需求事实锚点、实现记录、测试记录、构建记录、HTTP 验收记录 |
| S10 验收发布 | `formal-change` 必须 validate/archive；`structure-only` 不伪造归档 | `finishing-a-development-branch`（需要合并/PR时）+ `verification-before-completion` | 无 | 关闭前必须完成 Jar、HTTP、S9 和人工锚点 |

事件触发技能（均为会话级外部 Skill，不属于 `docs/.workflow/skills/` 项目内 Skill）：
- 收到人工或外部 code review 反馈：使用 `receiving-code-review`，先验证再改。
- 需要发起代码审查：使用 `requesting-code-review`。
- 上下文接近阈值或阶段切换：使用 `strategic-compact`，但不得替代上下文包。

> 说明：feature 线把 `workflow_mode` 作为流程深度（`standard` / `lightweight`），把 `execution_mode` 作为执行方式（`agentic` / `fallback`）；`input_mode` 只区分 `doc / idea` 的入口来源，不改变 S1-S10 的阶段数量和门禁顺序。

> 说明：本项目的 Skills 分两类。`superpowers` / `writing-plans` / `strategic-compact` 这类是会话级外部执行技能，由运行环境提供；`docs/.workflow/skills/feature/` 和 `docs/.workflow/skills/bugfix/` 是项目内 workflow Skills 包。后者必须是自包含目录，目录内至少包含 `SKILL.md`，可按需配 `references/`、`scripts/`、`assets/`，并遵循“先最小读取、再渐进展开”的原则。

### 1.5 Git 与文档边界（强制）

- `docs/` 下的需求、方案、计划、实现记录、测试记录、上下文包和 OpenSpec 决策记录是项目工作流文档，默认只落在当前项目文档目录，**大模型不得自动 `git add`、暂存或提交这些文档**。
- 文档是否纳入 Git 由人工决定；若项目 `.gitignore` 忽略 `docs/`，这是允许状态，不视为流程失败。
- S6 及后续涉及代码修改的阶段，代码必须在 `using-git-worktrees` 创建的代码 worktree 中修改。主项目目录作为文档根和状态机根，代码 worktree 作为代码实现根。
- 若 feature 明确为 docs-only（只修改需求/方案/流程规范/Agent/Skill/模板/验证文档，不修改 `src/`、`pom.xml`、运行配置或业务测试），可在实现记录写明 docs-only 范围后执行 `auto docs-only-worktree-not-required`。该动作只把 `worktree_created` 作为“不需要代码 worktree”门禁满足，`workflow_runtime.code_worktree_path` 必须保持为空，并写入 `exception_log`；不得用于任何业务代码变更。
- 子Agent 或 fallback 主Agent 写代码时，必须明确两个根路径：`doc_root=<主项目目录>`、`code_root=<git worktree目录>`。工作流脚本、`state.json`、`04-实现记录/` 等文档产物写入 `doc_root`；`src/`、`pom.xml`、测试代码等实现产物写入 `code_root`。
- 禁止在代码 worktree 中新建一套 feature 文档，避免分支树与当前文档树状态分裂。

## 2. 主Agent + 子Agent 架构

### 2.1 主Agent 职责

```
决策 / 指挥 / 检查 / 修正
严格按规范，不主观判断，有疑问才问人
```

**不做**：替代子Agent长期串行执行、跳过检查、无记录推进
**做**：读 state.json → 决定下一步 → 记录 step-start → 派遣子Agent → 检查输出 → 通过则记录 step-done

### 2.1.1 多Agent硬性原则

- 标准流程必须使用多Agent：需求交叉验证、技术方案、落地计划、任务拆分、任务实现、规格符合性复核、代码质量复核、测试验证、HTTP接口验收、全链路验证分别由对应子Agent执行或复核。
- 进入 S2/S3/S6/S7/S8 前，主Agent 必须确认当前运行环境是否已获得“可派遣子Agent/并行Agent”的明确授权。
- 已授权：必须派遣子Agent，并把派遣、返回、修正记录写入 `subagent_log`。
- 未授权或平台不可用：不得静默改为主线串行执行，必须先记录 `exception subagent-fallback "原因"`，再按同一检查清单由主Agent临时执行。
- `exception subagent-fallback` 只能追加主线 fallback 动作，不得覆盖当前 `allowed_next_actions` 中已有的精确步骤或人工锚点。
- 主线 fallback 是例外，不是常态；同一 feature 若连续两个阶段 fallback，必须在恢复包风险项中提示人工确认是否继续。

### 2.2 子Agent 列表

每个子Agent 在 `docs/.workflow/agents/` 下有独立定义文件：

| 子Agent | 触发阶段 | 输入 | 输出 |
|---------|----------|------|------|
| 需求交叉验证 | 需求确认后 | 需求输入文档 + 需求确认文档 | 差异报告.json + 需求事实锚点.json |
| 技术方案设计 | 需求基线冻结后 | 需求确认基线 + 需求事实锚点 | 技术方案.md + 需求方案映射.json + 代码影响点与依赖逻辑清单.md + 技术方案一致性检查.json |
| 落地计划 | 技术方案冻结后 | 需求基线 + 技术方案 | 落地计划-v1.md |
| 任务拆分 | 落地计划完成后 | 落地计划 + 需求/方案映射 | 任务清单.json + RDTV映射表.json |
| 任务实现 | 每个T任务 | T编号 + 方案 + 需求基线 | 代码 + 实现记录.md |
| 规格符合性复核 | 每个T实现后 | 当前T上下文 + 实现记录 + 代码diff | 规格复核记录.md |
| 代码质量复核 | 规格复核通过后 | 当前T代码 + 测试 + 规格复核记录 | 质量复核记录.md |
| 测试验证 | 实现完成后 | 实现记录 + 验收口径 | 测试记录.md + V编号 |
| HTTP接口验收 | 单元测试通过并打包后 | Jar包路径 + 启动信息 + API验收清单 | HTTP验收记录.md |
| 全链路验证 | 测试通过后 | RDTV完整映射表 | RDTV报告.json |

### 2.3 主Agent工作协议（强制）

**派遣子Agent前**：
- 读 state.json 确认阶段和允许动作
- 确认子Agent输入文档版本正确
- 先生成或更新 `06-上下文包/上下文包-<阶段>.md`，只传最小必要上下文和精确文件路径
- 调用 `stage_gates.py subagent-start <FID> <子Agent名> <派遣JSON>` 记录派遣
- 下达明确指令，限定范围，禁止自由发挥

**子Agent返回后**：
- 按子Agent定义文件中的检查清单逐项核查
- 调用 `stage_gates.py subagent-done <FID> <子Agent名> <结果JSON>` 记录返回摘要；并行或同名子Agent返回时必须携带 `subagent-start` 输出的 `dispatch_id`
- 发现问题 → 指令子Agent修正，记录修正原因
- 全部通过 → 调用 step-done 记录关键结论
- 修正3次仍不通过 → 触发 approve-correction 人工锚点

**每个小步骤的强制纪律**：
- 开始前必须调用 `step-start`，且 `goal`、`expected_outputs`、`done_definition`、`next_step` 必须完整非空
- 小步骤尚未完成但已完成一个可检查动作时，必须调用 `progress`
- 完成后必须调用 `step-done`
- 不允许多个小步骤做完后批量补记
- 若中断发生在 `step-start` 与 `step-done` 之间，恢复时必须检查该步骤是否已部分完成

缺少完整 `step-start` 的风险：
- 恢复时只知道“正在做哪个步骤”，不知道目标、完成条件和下一步，只能靠模型推测。
- 容易重复执行、漏执行，或把未完成步骤误判为完成。
- `expected_outputs` 缺失时无法检查产物是否存在，恢复确认卡会失去客观依据。

**每个T任务完成后**（主Agent锚点，不是人工锚点）：
- 检查：实现记录完整？偏离检测通过？测试通过？
- 通过 → 自动进入下一个T
- 不通过 → 指令子Agent修正
- 不确定 → 人工确认
- `implement-Txx` 执行完成前必须已增量更新 `04-实现记录/*.md`，并在 `step-done` 的 `outputs` 中列出该实现记录。
- 实现记录必须提及当前 T 编号、修改/复用文件、关键取舍、验证命令和结果。
- 不允许多个 T 全部完成后再集中补实现记录。

实现记录不按 T 增量更新的风险：
- 中断恢复时只能从代码 diff 猜实现意图，无法判断当前 T 是否真的完成。
- 实现偏离校验缺少证据，可能误报、漏报或放过超范围修改。
- R-D-T-V 链路会暂时断裂，后续测试和全链路验证可能建立在不完整事实上。
- 需求事实锚点与技术方案一致性检查断裂时，后续实现会在正确外形下继续跑偏。

### 2.4 上下文隔离原则

- 子Agent 拥有独立上下文窗口
- 主Agent 只接收子Agent 的结构化摘要，不接收中间过程
- 主Agent 上下文用量超过 `docs/.workflow/project_config.json` 的 `workflow.context_compact_threshold` 时主动 `/compact`
- 阶段切换前必须 `/compact`，并指定保留：state.json 内容、关键结论、下一步动作

## 3. 双轨入口

### 3.1 入口判断

```
有具体需求文档（PRD/设计稿/说明书）→ 轨道 A
只有想法/口述描述                    → 轨道 B
```

### 3.2 轨道 A — 已有需求文档

```
A1. 自动执行 init_feature.py 建骨架
A2. 文档放入 00-需求输入/
A3. 主Agent 与人工讨论需求输入（脑暴展开，确认理解）

【人工锚点 ① approve-req-input】
    人工深度参与确认"需求输入脑暴完整"

A4. 主Agent 启动 OpenSpec 结构化提炼
A5. 主Agent 与人工讨论需求确认（逐条检查细节、逻辑）
A6. 主Agent 派遣"需求交叉验证"子Agent
    对照需求输入 vs 需求确认，多视角找遗漏和矛盾

【人工锚点 ② approve-req-final】
    人工查看交叉验证报告，确认"需求无误、验证通过"
    需求基线正式冻结
```

### 3.3 轨道 B — 只有想法

```
B1. 自动执行 init_feature.py 建骨架（input_mode=idea）
B2. 主Agent 使用 `superpowers:brainstorming` 技能
    与人工脑暴讨论
B3. 讨论结论由主Agent写入 00-需求输入/

【人工锚点 ① approve-req-input】
    人工确认"想法已被完整捕捉"

B4. 之后与轨道 A 的 A4-A6 完全相同

【人工锚点 ② approve-req-final】
```

## 4. 主流程

### 4.1 完整阶段序列

```
init
  ↓ approve-req-input（锚点①）
需求确认
  ↓ approve-req-final（锚点②）→ 需求基线冻结
技术方案
  ↓ 自动转移（主Agent判断有疑问则触发锚点③）
落地计划
  ↓ 自动转移
任务拆分
  ↓ 映射闭环校验通过自动转移
实现（每个T 都有主Agent锚点）
  ↓ 所有T完成
测试验证
  ↓ 自动转移
构建验收
  ↓ 打 Jar 包
  ↓ 人工本地启动部署 + HTTP API 验证
交叉验证（全链路）
  ↓ approve-verify（锚点④）
验收发布
  ↓ approve-release（锚点⑤）
done
```

### 4.2 阶段详细定义

---

**S1 — 需求输入（人工深度参与）**

- 目的：脑暴发散，把想法或文档充分展开
- 主导：人工 + 主Agent 对话
- 自动校验：无（这一阶段重在人工讨论）
- 出口：锚点 ① `approve-req-input`

---

**S2 — 需求确认（人工深度参与 + 多Agent交叉验证）**

- 目的：逐条检查细节、逻辑、完整性
- 工具链：
  1. OpenSpec 完整性回扫
  2. 记录 OpenSpec 使用模式：`structure-only` 或 `formal-change`
  3. 主Agent 与人工逐条确认
  4. 子Agent A（需求交叉验证）：对照需求输入 vs 需求确认，并抽取需求事实锚点
- 自动产出：R编号体系、OpenSpec决策记录、覆盖检查.json、差异报告.json、需求事实锚点.json
- 自动校验：需求完整性校验器；需求交叉验证通过前必须确认 `01-需求确认/需求事实锚点.json` 已存在，否则不得触发 `req-cross-validated`
- 出口：锚点 ② `approve-req-final`

**S3 — 技术方案（主Agent主导，有疑问才问人）**

- 目的：根据需求基线生成技术方案
- 工具链：
  1. 主Agent 派遣"技术方案设计"子Agent
  2. 子Agent 输出方案草稿 + D编号 + 需求方案映射 + 代码影响点与依赖逻辑清单 + 技术方案一致性检查
  3. 主Agent 先检查需求事实是否被完整继承，再检查映射完整性；发现不确定项 → 触发锚点 ③
- 硬门禁：需求事实锚点必须被继承；每条R必须有对应D
- 自动校验：技术方案事实继承校验器、R→D 映射校验器
- 出口：自动进入 S4，除非锚点 ③ 触发

**锚点 ③（条件触发）**`approve-tech-question`
- 由主Agent判断是否触发
- 触发时机：技术选型不确定、架构方案有多种、需要权衡取舍

**S4 — 落地计划**

- 工具链：主Agent 结合 Superpowers writing-plans 方法派遣"落地计划"子Agent
- 输出：`03-落地计划/落地计划-v1.md`（实施顺序、依赖、风险、回滚方案）
- 小步纪律：

  2. 产出落地计划文档
  3. 再执行 `step-done write-implementation-plan`
  4. 最后执行 `auto plans-complete`
- 自动出口：`plans-complete` 后进入 S5

**S5 — 任务拆分**

- 工具链：主Agent 派遣"任务拆分"子Agent 生成 T 编号体系
- 输出：`03-落地计划/任务清单.json` + `03-落地计划/RDTV映射表.json`（R→D→T 已填，V 待填）
- 小步纪律：
  1. 先执行 `step-start write-task-split`
  2. 产出任务清单和 RDTV 初版
  3. 再执行 `step-done write-task-split`
  4. 执行 `validators.py rdt_mapping <FID>`
  5. 校验通过后由校验器触发 `auto rdt-mapping-passed`
- 硬门禁：映射闭环校验器
  - 有T无R / 有T无D / T无完成定义 / T无验收方式 → 标红阻断
- 自动出口：校验通过自动进入 S6

**S6 — 实现（每个T都有主Agent锚点）**

- 工具链：
  1. 主Agent 使用 `using-git-worktrees` 创建代码 worktree，并确认 `git worktree list` 中存在独立 worktree。文档仍写入主项目 `docs/01-features/Fxx/`。
  2. 主Agent 使用 `subagent-driven-development` 的调度纪律：每个 T 派遣一个 fresh "任务实现"子Agent；随后依次派遣 `规格符合性复核`、`代码质量复核`；一个 T 完全关闭后再进入下一个 T。
  3. 子Agent 使用 `test-driven-development`：先写当前 T 的失败测试并验证 RED，再写最小实现通过 GREEN，最后重构并保持测试通过。
  4. 每个 T 完成前，子Agent 必须增量写 `04-实现记录/实现记录-YYYYMMDD-Txx.md`，记录 RED/GREEN 证据、修改范围、D 字段对照和 acceptance 验证结果。
  5. `规格符合性复核` 先检查是否严格满足 R/D/T 和 acceptance；未通过时返回原 `任务实现` 子Agent 修正。
  6. 规格复核通过后，`代码质量复核` 再检查实现质量、测试质量和编码纪律；未通过时返回原 `任务实现` 子Agent 修正。
  7. 主Agent 检查该 T 的实现记录是否已写入、提及当前 T 编号，并包含 TDD RED/GREEN 证据。
  8. 主Agent 运行实现偏离检测器和聚焦测试。
  9. 规格复核通过 + 质量复核通过 + 偏离检测通过 + 测试通过 → 主Agent 锚点 → 进入下一个T。
  10. 任一复核或检测不通过 → 指令原 `任务实现` 子Agent修正。
- S6 技能职责：
  - `using-git-worktrees`：只解决代码隔离，不替代分支策略、测试或子Agent调度。
  - `subagent-driven-development`：解决“每个 T fresh 子Agent + 规格复核 + 质量复核”的执行纪律；当前流程默认串行执行实现 T，禁止并行派遣多个实现子Agent。
  - `test-driven-development`：解决“先证明测试会失败，再实现”的验证纪律；没有 RED 证据的实现不能关闭 T。
  - `executing-plans`：仅在没有子Agent能力时作为降级执行纪律；降级后仍必须在代码 worktree 中执行、仍必须 TDD、仍必须写实现记录和跑同样门禁。
- 物理门禁：worktree 创建失败 → 流程停止
- 例外：`exception worktree "原因"` 只记录阻塞原因，不等同于 worktree 已创建，不允许因此开始写代码；除非人工明确改变“代码必须在 worktree 中实现”的项目约束。
- docs-only 例外：仅当实现记录确认本 feature 不涉及业务代码、构建配置或运行配置时，可执行 `auto docs-only-worktree-not-required` 替代 `auto worktree-created`，继续 S6 任务实现与后续验证。
- 子Agent 例外：若当前环境未授权派遣子Agent，必须先记录 `exception subagent-fallback "原因"`，再由主Agent按同一 T 清单执行。
- 硬门禁：`implement-Txx` 的 `step-done` 必须同时列出实现记录、规格复核记录、代码质量复核记录；实现记录必须提及当前 T 编号和 TDD RED/GREEN 证据，两类复核记录必须明确 `passed: true/通过`。

**S7 — 测试验证**

- 工具链：子Agent D（测试验证）
- 输出：测试记录-YYYYMMDD.md + V 编号
- 例外：若当前环境未授权派遣子Agent，先记录 `exception subagent-fallback "原因"`，再由主Agent按同一测试清单执行并记录。
- 自动出口：`test-done` 后进入 S8 构建验收

**S8 — 构建验收（构建产物 + HTTP API）**

- 目的：单元测试/聚焦测试通过后，生成真实运行产物，并在人工本地启动后由大模型执行 HTTP API 功能验证。
- 工具链：项目构建命令 + 人工本地启动/部署 + HTTP接口验收子Agent。Java/Maven 项目默认使用 `mvn -DskipTests package` 和 `target/*.jar`，非 Java 项目必须覆盖 `docs/.workflow/project_config.json`。
- 强制步骤：
  1. 主Agent 执行 `step-start package-artifact`（Java 项目可继续使用 `package-jar`）。
  2. 执行 `docs/.workflow/project_config.json` 中声明的 `build_command`；Java/Maven 默认是 `mvn -DskipTests package`。
  3. 将产物路径、构建命令、构建结果写入 `05-测试验证/构建记录-YYYYMMDD.md`。
  4. 打包成功后执行 `auto artifact-package-done`。
  5. 主Agent 生成 `05-测试验证/HTTP验收清单-YYYYMMDD.md`，明确 URL、方法、请求体、期望状态码、期望响应字段和副作用检查。
  6. 主Agent 提示人工本地启动/部署服务，并明确 profile、端口、依赖服务、测试数据前提。
  7. 人工确认服务已启动后，派遣"HTTP接口验收"子Agent，由大模型执行真实 HTTP 请求。
  8. HTTP 验收记录写入 `05-测试验证/HTTP验收记录-YYYYMMDD.md`。
  9. HTTP 验收通过后执行 `auto http-acceptance-done`。
- docs-only 例外：若 feature 只修改文档、流程规范、Agent、Skill 或模板，且无运行产物和 HTTP 接口影响，S8 仍必须保留阶段但可使用替代门禁：
  1. 写入 `05-测试验证/构建记录-YYYYMMDD.md`，明确 `not_applicable: true`、docs-only 范围、替代验证命令和 `passed: true`。
  2. 执行 `auto docs-only-artifact-not-required`。
  3. 写入 `05-测试验证/HTTP验收清单-YYYYMMDD.md` 与 `05-测试验证/HTTP验收记录-YYYYMMDD.md`，明确 HTTP 不适用原因、替代检查项和 `passed: true`。
  4. 执行 `auto docs-only-http-acceptance-not-required`。
- 硬门禁：
  - 构建产物未生成或路径未记录，禁止进入 HTTP 验收；默认 Java/Maven/Jar 配置可由 `project_config.json` 覆盖。
  - HTTP 验收未通过，禁止进入全链路交叉验证。
  - docs-only 替代门禁不得用于任何业务代码、构建配置、运行配置或接口行为变更。

---

**S9 — 交叉验证（全链路）**

- 工具链：
  1. 子Agent E（全链路验证）多视角并行
  2. OpenSpec 一致性复核
  3. RDTV 闭环校验器
- 输出：RDTV报告.json + 结构化验证报告
- 例外：若当前环境未授权派遣子Agent，先记录 `exception subagent-fallback "原因"`，再由主Agent执行同等全链路验证。
- 硬门禁：每条 R 至少 1 个 V，V 结果不能为空；HTTP 验收结论必须纳入全链路验证报告；需求事实锚点中的 `must_preserve` 事实必须全部通过技术方案一致性检查
- 出口：锚点 ④ `approve-verify`

---

**S10 — 验收发布**

- 目的：全链路验证和 HTTP 验收均通过后，关闭需求。
- 工具链：OpenSpec change 归档 + 发布关闭确认
- 强制步骤：
  1. 确认 `artifact_package_done`、`http_acceptance_done`、`cross_validate_done`、`rdtv_mapping_complete` 均为 true。
  2. 检查 S2 的 OpenSpec 决策记录：
     - `formal-change`：对 feature-local OpenSpec change 做一致性检查；如使用根目录临时镜像执行 `openspec validate <change-id> --strict`，通过后将归档结果回写到 `docs/01-features/Fxx/openspec/变更归档/`，并清理或标注根目录镜像不是权威文档。
     - `structure-only`：确认 `OpenSpec决策记录` 已说明“不需要正式 change”的原因，不创建伪归档。
  3. 人工锚点 ⑤ `approve-release`。
- 硬门禁：未完成 `artifact_package_done` 或 `http_acceptance_done` 时，禁止执行 `approve-release`。
- 出口：锚点 ⑤ `approve-release` → done

## 5. 人工锚点汇总

| 锚点 | 类型 | 时机 | 口令 |
|------|------|------|------|
| ① | 固定 | 需求输入讨论完成 | `approve-req-input` |
| ② | 固定 | 需求确认 + 交叉验证完成 | `approve-req-final` |
| ③ | 条件 | 技术方案有不确定项时主Agent暂停 | `approve-tech-question` |
| ④ | 固定 | 全链路交叉验证完成 | `approve-verify` |
| ⑤ | 固定 | 人工本地部署后的 HTTP API 验收通过 | `approve-release` |

**其他口令**：
- `exception-[类型] [原因]` — 记录例外操作
- `approve-correction` — 子Agent 多次修正仍不通过时人工介入
- `exception subagent-fallback "原因"` — 当前运行环境不能派遣子Agent 时，允许主Agent按同一检查清单在主线执行，并必须记录原因
- `exception worktree "原因"` — 只记录 worktree 创建阻塞，不自动放行写代码，也不把 `worktree_created` 置为 true

**T任务锚点**：每个T完成后由主Agent内部决策，不需要人工介入，除非主Agent判断"不确定"。

## 6. 自动完整性校验体系

输出全部为结构化 JSON，不是自然语言。校验未通过 = state 物理上不转移。

| 校验器 | 触发时机 | 不通过效果 |
|--------|----------|-----------|
| OpenSpec 决策记录 | S2 结构化后 | state 不转移，禁止冻结需求基线 |
| 需求完整性 | S2 结束 | state 不转移，输出缺失项 |
| R→D 映射 | S3 结束 | state 不转移，输出缺口 |
| R→D→T 映射闭环 | S5 结束 | state 不转移 |
| 实现偏离 | 每次实现记录写入后 | 写入 exception_log |
| Jar 包构建 | S8 打包结束 | state 不转移 |
| HTTP API 验收 | S8 结束 | state 不转移 |
| RDTV 闭环 | S9 结束 | state 不转移 |

实现偏离校验口径：
- 对当前 T，必须在实现记录中提及该 T 的每个 `scope`，即说明“已修改”或“复用/未修改”。
- 若实现记录按整个 feature 汇总，疑似超范围文件必须按所有 T 的 `scope` 并集判断，不能把其他 T 的合法文件误判为当前 T 偏离。
- 若确有超出所有 T scope 的文件，必须补充任务清单或记录例外原因。

## 7. 会话恢复机制（双粒度）

### 7.1 阶段级（每次锚点通过后）

- state.json 自动快照（保留最近10个）
- 恢复包.md 自动更新恢复确认卡
- 导航.md 自动更新当前阶段

### 7.2 步骤级（每开始 / 进展 / 完成一小步后）

主Agent 在开始一小步前必须调用：
```bash
python3 docs/.workflow/scripts/stage_gates.py step-start <FID> <步骤名> <计划JSON>
```

计划 JSON 最少包含：
```json
{
  "goal": "这一步要达成什么",
  "expected_outputs": ["预期输出路径"],
  "done_definition": ["完成判定1", "完成判定2"],
  "next_step": "下一步名称"
}
```

脚本必须强校验：
- `goal` 非空。
- `expected_outputs` 非空；无文件输出时也要写明状态更新、验证命令输出或人工确认记录。
- `done_definition` 非空。
- `next_step` 非空。
- `human_approval_pending=true` 时禁止 `step-start`，必须先执行当前人工锚点。
- `step_name` 命中 `blocked_actions` 时禁止 `step-start`；匹配大小写不敏感，`-` 与 `_` 视为等价。
- 不允许传纯文本计划，不允许空数组，不允许先执行后补 `step-start`。

作用：
- 明确当前正在执行的步骤
- 把预期输出和完成定义写入 `state.json`
- 一旦中断，恢复时先检查“进行中步骤”是否已部分完成

当当前步骤尚未完全完成，但已完成一个可检查的小动作时，主Agent 必须调用：
```bash
python3 docs/.workflow/scripts/stage_gates.py progress <FID> <进展JSON>
```

进展 JSON 最少包含：
```json
{
  "completed_action": "已完成的具体动作",
  "key_conclusions": ["客观结论1"],
  "outputs": ["本小动作新增或更新的路径"],
  "verification": ["已执行的验证命令"],
  "next_step": "下一小动作"
}
```

脚本必须强校验：
- 当前存在未闭合的 `step-start`。
- `completed_action` 非空。
- `key_conclusions` 非空。
- `next_step` 非空。
- `outputs` 与 `verification` 不能同时为空，否则恢复时没有可检查证据。
- 不允许传纯文本进展。

`progress` 只更新 `current_step_log.progress_notes` 和 `in_progress_step.last_progress`，不得关闭当前步骤，不得改变阶段，不得把未完成的大步骤标记为完成。

当前步骤满足完成定义后，主Agent 必须调用：
```bash
python3 docs/.workflow/scripts/stage_gates.py step-done <FID> <步骤名> <结论JSON>
```

脚本必须强校验：
- 存在同名且未闭合的 `step-start`。
- 当前 `in_progress_step.step` 与本次 `step-done` 的步骤名一致。
- `key_conclusions` 非空。
- `next_step` 非空；即使下一步是自动门禁，也必须写明例如 `auto test-done`。
- 不允许传纯文本结论，不允许事后批量补记。

写入 `state.json` 的 `current_step_log`：
```json
{
  "started_steps": [
    {
      "step": "openspec-analysis",
      "stage": "需求确认",
      "started_at": "2026-05-15T14:10:00",
      "goal": "完成需求结构化与R编号抽取",
      "expected_outputs": ["01-需求确认/需求说明书-v1.md"],
      "done_definition": ["需求说明书生成", "R编号完整"],
      "next_step": "coverage-check",
      "status": "in_progress"
    }
  ],
  "progress_notes": [
    {
      "step": "openspec-analysis",
      "stage": "需求确认",
      "recorded_at": "2026-05-15T14:20:00",
      "completed_action": "已完成R编号初稿",
      "key_conclusions": ["识别R01-R08"],
      "outputs": ["01-需求确认/需求说明书-v1.md"],
      "verification": [],
      "next_step": "覆盖检查"
    }
  ],
  "completed_steps": ["openspec-analysis", "superpowers-clarify"],
  "pending_steps": ["coverage-check"],
  "key_conclusions": [
    "识别隐含需求R06/R07/R08",
    "确认不支持第三方登录",
    "验收口径：POST /api/login 200 + token 24h"
  ]
}
```

同时写入顶层：
```json
{
  "in_progress_step": {
    "step": "openspec-analysis",
    "stage": "需求确认",
    "started_at": "2026-05-15T14:10:00",
    "goal": "完成需求结构化与R编号抽取",
    "expected_outputs": ["01-需求确认/需求说明书-v1.md"],
    "done_definition": ["需求说明书生成", "R编号完整"],
    "next_step": "coverage-check",
    "status": "in_progress"
  }
}
```

### 7.3 关键结论格式（规范化）

```json
{

  "completed_at": "2026-05-15T14:32:00",
```

字段全部规范化，不允许自由发挥。

### 7.4 新会话启动协议（5步，硬性）

如收到新 Bug 描述（含“报错/异常/错误/导出失败/不准/崩了”等特征词且非 BFxx 格式），必须先用 AskUserQuestion 向用户确认“是否启动自动排查流程 BFxx”；仅在用户明确回复“继续/确认/yes”后才允许调用 `python3 docs/.workflow/scripts/init_bugfix.py "<问题描述>"` 初始化骨架；不得静默创建目录或文档。

```
Step 1. 读 state.json
        → 确认 current_stage / current_step / completed_steps / pending_steps
        → 确认 in_progress_step
        → 读 key_conclusions

Step 2. 读 state.baseline 中的基线文档，验证版本一致

Step 3. 检查 in_progress_step 是否已部分完成
        → 检查 expected_outputs 是否存在
        → 对照 done_definition 判断是否满足完成条件
        → 记录 recovery_review 结果

Step 4. 对照规范检查当前阶段是否完成且正确
        → 未完成 → 从断点继续
        → 完成但有问题 → 询问人工

Step 5. 输出恢复确认卡（固定模板，10行以内）

Step 6. 等待指令，不允许提前行动
```

## 8. 上下文管理规范（最佳实践）

### 8.1 四层防护

| 层 | 机制 | 责任方 |
|----|------|--------|
| 第一层 | 上下文包 + 精确路径输入 | 主Agent 生成，子Agent 只按需读取 |
| 第二层 | 子Agent 隔离 | 架构层面，自动生效 |
| 第三层 | 阶段切换前 /compact | 主Agent 主动 |
| 第四层 | 上下文超过配置阈值后主动压缩 | 主Agent 监控 |

### 8.1.1 上下文包机制（优先于 /compact）

每个阶段开始前，主Agent 必须在 `06-上下文包/` 下生成或更新一个阶段上下文包：

```text
06-上下文包/上下文包-S6-实现.md
06-上下文包/上下文包-S7-测试验证.md
06-上下文包/上下文包-S8-构建验收.md
```

生成命令：

```bash
python3 docs/.workflow/scripts/context_packets.py build <FID> S6 --task T01
python3 docs/.workflow/scripts/context_packets.py build <FID> S8
python3 docs/.workflow/scripts/context_packets.py list <FID>
```

`context_packets.py` 会更新 `state.json.context_manifest.current_packet`，恢复时必须检查该字段是否与当前阶段匹配。`stage_gates.py subagent-start` 会校验 `context_packet` 是否存在、是否等于 `state.json.context_manifest.current_packet`、是否登记在 `context_manifest.packets`，以及该包阶段是否等于当前 `state.current_stage`；不一致则禁止派遣。

上下文包只允许包含：
- 当前 `state.json` 的关键字段摘要：阶段、步骤、合法下一动作、基线路径、checklist。
- 当前阶段必须使用的文档路径，不复制全文。
- 当前 T 或当前 V 的 JSON 片段，不复制整个任务清单。
- 关键结论和人工确认结论。
- 需要读取的代码文件精确路径清单。
- 禁止事项和本阶段完成定义。

上下文包禁止包含：
- 大段原始需求全文。
- 全量代码粘贴。
- 全量测试日志。
- 与当前阶段无关的历史讨论。

子Agent 收到任务时默认只读上下文包和其中列出的精确路径；需要额外文件时必须说明原因，并优先用 `rg` 定位后按片段读取。

### 8.2 阶段切换 /compact 指令

阶段转移前主Agent 必须执行：

```
/compact 保留以下内容：
- 当前 state.json 完整内容
- 当前基线文档路径
- 最近3条 key_conclusions
- 下一阶段 allowed_next_actions
压缩其余对话历史
```

### 8.3 子Agent 派遣原则

- 子Agent 名字"任务形状"（如 `需求交叉验证`、`技术方案设计`），不要泛化命名
- 工具权限最小化（只读类只给 Read/Grep，编码类才给 Write）
- 主Agent 接收的是子Agent 的结构化摘要，不是中间过程
- 子Agent 使用平台可用的执行型模型；主Agent 使用平台可用的最强决策模型。迁移到不同平台时以角色职责为准，不硬编码模型名称。
- 派遣前必须执行 `subagent-start` 记录：子Agent 名称、上下文包路径、输入路径、预期输出路径
- 返回后必须执行 `subagent-done` 记录：摘要、输出路径、关键结论、修正次数；并行或同名子Agent返回时必须携带 `dispatch_id`
- `subagent-start` 必须提供非空 `context_packet`、`input_paths`、`output_paths`、`instruction`；上下文包必须存在、已登记到 `state.json.context_manifest`，且阶段必须等于当前 `state.current_stage`。
- `subagent-start` 的 `input_paths` 必须真实存在；`output_paths` 是预期产物路径。
- 如果子Agent定义文件 frontmatter 声明 `prerequisites`，`subagent-start` 必须先校验其中的 `path` 或 `glob` 是否存在；缺失时给出明确前置缺口并拒绝派遣。
- `subagent-start` 会生成并打印 `dispatch_id`；`subagent-done` 必须携带该值并按 `dispatch_id` 精确匹配，缺失、为空或找不到未关闭派遣记录时脚本必须失败。
- `subagent-done` 必须匹配一个未关闭的 `subagent-start`；没有派遣记录时禁止事后补返回记录。
- `subagent-done` 的 `output_paths` 必须真实存在，否则返回摘要不能作为完成证据。
- 当前步骤最近一次子Agent结果若为 `dispatched`、`failed`、`partial` 或 `blocked`，禁止把步骤标记为 `done`，也禁止执行 `auto` 或普通人工锚点；必须先重派并返回 `done`，或把当前步骤记录为 `failed/blocked` 后进入修正流程。`approve-correction` 不受该阻断限制。
- 未写入 `subagent_log` 的派遣视为未发生；若平台无法派遣，必须写 `exception subagent-fallback`
- 编码类子Agent 一次只处理一个 T；当前流程遵循 `subagent-driven-development`，默认串行派遣实现子Agent。`parallel_group` 只用于识别理论并行性和后续优化，不作为当前流程并行派遣许可。
- `artifact-package-done` 会按 `docs/.workflow/project_config.json` 校验构建产物与 `05-测试验证/构建记录-*.md`；默认配置为 Java/Maven/Jar
- `project_config.json` 的 `workflow` 段声明上下文提醒阈值、压缩阈值、子Agent修正阈值，以及 feature/bugfix 流程启用开关；`stage_gates.py` 必须读取这些配置，不得把阈值散落硬编码。
- `http-acceptance-done` 会校验 `artifact_package_done`、`05-测试验证/HTTP验收记录-*.md` 和 `passed: true/通过` 结论
- `docs-only-artifact-not-required` 与 `docs-only-http-acceptance-not-required` 只适用于无运行产物、无 HTTP 行为影响的文档/规范/Agent/Skill/模板变更，必须有对应构建记录、HTTP验收清单和 HTTP验收记录说明 `not_applicable: true` 与替代验证证据。

### 8.4 上下文健康度报告

每次 step-start / step-done 都附带：

```json
{
  "context_usage_pct": 45,
  "compact_recommended": false
}
```

`compact_recommended: true` 时，下一步开始前必须先 `/compact`。

## 11. 跨项目迁移指南

迁移到新项目只需以下动作：

1. 复制 `workflow-kit/docs/` 到目标项目。
2. Codex 项目复制 `workflow-kit/开发AGENTS.md` 为目标项目 `AGENTS.md`；若已存在则合并。
3. Claude Code 项目复制 `workflow-kit/开发CLAUDE.md` 为目标项目 `CLAUDE.md`；若已存在则合并。
4. 复制 `INSTALL.md` 和 `使用指南.md` 供人工查阅。
5. 不迁移来源项目已有的 `docs/01-features/Fxx-*`、`docs/02-bug-fix/` 具体记录、`docs/03-knowledge/` 业务内容、业务版 `CLAUDE.md` 或根目录 `openspec/` 历史 change。

无需硬编码项目路径。脚本通过 `Path(__file__).parent` 自动定位，不依赖 cwd 或 .git。

迁移后第一件事：

```bash
python3 docs/.workflow/scripts/init_feature.py --list
```

若新项目尚无 feature，`--list` 应正常返回空列表。开始第一个需求时再用 `init_feature.py` 创建，脚本会自动创建 `06-上下文包/` 和 `context_manifest`。

Obsidian vault 指向项目根目录，零配置实时同步。

## 12. 修正失败的处理

子Agent 修正 3 次仍不通过时：

1. 主Agent 写入 `exception_log`
2. 触发 `approve-correction` 人工锚点
3. 人工判断是否需要：
   - 调整 T 任务定义
   - 回退到上一阶段
   - 跳过该项并标记
4. 不允许主Agent 自行决定

脚本在 `subagent-done` 收到 `corrections` 数组长度 `>= 3` 时，会自动写入 `exception_log` 的 `correction-threshold` 记录并提示 `approve-correction` 命令；不会自动改变阶段或 allowed actions。

若子Agent返回 `failed`、`partial` 或 `blocked`，该结果本身会写入 `subagent_log`，但不能作为完成证据。主Agent只能选择：补充输入后重派同一阶段子Agent并获得 `done`；或把当前步骤以失败/阻塞状态关闭，随后触发 `approve-correction` 由人工决定回退、调整任务或继续。
